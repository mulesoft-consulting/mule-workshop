Module 1: Design Center

Anypoint Platform provides first-class tooling to support the needs of the API Designer. Anypoint Design Center provides a robust RAMl editor for designing the specifications of the API and standing up a "mock" service to accelerate development efforts. It is also tightly integrated with Anypoint Exchange to simplify the process of discovering and referencing other API design-oriented assets like common data types, traits, schemas, and data examples.

For the purposes of this workshop we will design new API’s using REST. RESTful API’s are very popular right now and support a variety of patterns that help solve many reliability, scalability and integration challenges you might face. We will use an API-first design approach using the RESTful API Modeling Language (RAML) standard. We will follow REST best practices to promote adoption to your consumers. This lab will also introduce the main concepts of resource oriented design and will show how RAML is used to bring APIs to life.

For more information on RAML, please see: www.raml.org.

Step 1: Create an API project in Design Center

Anypoint Design Center is used to design your API’s. When you arrive at the Design Center landing page you will see a list of API’s that have already been designed by your organization. In this step, you will create an API and design it using the Anypoint Design Center.

  1. You can access Anypoint Design Center from the home page of Anypoint Platform:

  1. To create a new API, Click on the Create button. Select "API Specification".

  1. Select "Create API Specification".

  1. Set the following values in the Add API form:
  • API name: <username> - Omni Channel Experience API.
  • Visual Editor: Select the Visual editor radio button.

Note: Since we are working in a shared organization, please give your API a unique name. In this exercise please prepend your API with your username.

  1. Click Create Specification.

  1. Let’s add some general details about the API.
  • Name the API by setting the title to <your_username> Omni Channel Experience API.
  • Since this is our first design let’s designate this API as version 1.0. Set version to 1.0
  • Set the media type to application/json.
  • Most API specifications will designate an API endpoint Base URI. This tells the API consumer where they will access the API. Set the URI to http://localhost:8081/api

You should now see the following in your API design:

Step 2: Create the Orders Resource

Now you need to add the /order resource.

  1. A resource is the definition of an entity that your API will manage. Click the + to the right of Resources, then choose Add resource.

  1. Rename the resource from /new-resource to /order
  2. Select the POST option and under documentation, give it a description: Creates a new Order instance as shown.

  1. Notice the generated RAML on the right.

You have completed creation of an /order resource with a basic description.

Step 3: Create the /order/{id}

In the previous step we created an Order collection resource and POST method to enable the e-commerce application to create a new Order instance.

What if we wanted to let the e-commerce application get the status of a single order?

We need to create a new resource underneath the Order instance and use the GET method.

  1. Hover the mouse cursor over the /order resource. You will notice that a + appears in the resource panel.

  1. Press the + button to create a new, nested resource.
  2. Notice that a default name of "new-resource" has been created.
  3. In addition, the GET method has been selected by default.

  1. Rename /order/new-resource to /order/{order_id}. Curly brackets indicate dynamic variables where a value will need to be passed when called.
  2. As we did in the previous step, enter a description: Order instance resource allowing to retrieve an order

Step 4: Specify data types for response

We need to create a new resource underneath the Order instance and use the GET method.

  1. Under the GET method click on the Responses tab
  2. Then click on the Add New Response button.

  1. You should now see a response with a 200 OK response code.

  1. Under the response, click Add Body to add a JSON response to the GET method.

  1. Specify the return type as Object

Step 5: Create an example for the data

We need to create a new resource underneath the Order instance and use the GET method.

  1. Let’s add an example JSON to the response. We need to expand the “bodies” section by clicking on “Details”

  1. Once expanded, you will see the “Example” section.

  1. Under “Example”, click the Edit tab and write some JSON code

  1. Example 1: create a very simple JSON

{  "order_id": "order_22" }

  1. Example 2: include return the order_id and a product list.

{

  "order_id": "order_22",

  "items": [

        { "product_id": "121", "name": "phone" },

        { "product_id": "122", "name": "charger" }

   ]

}

Step 6: Enable the Mocking Service

The API editor provides a mocking service, which allows users to interact with the API as if it was built and deployed. This feature allows you to rapidly iterate the API design with its consumer to finalize the contract.

  1. Look at the baseUri in the RAML Viewer. Notice it is set based on what we entered in the API Summary.
  2. Enable the mocking service by switching the Mocking Service to ON on top of the RAML viewer:

  1. After a few seconds, the mocking service will start, creating a new URL for the baseUri:

A very useful feature during the design phase is having the ability for your consumers to interact with your API without having to code anything.

Step 7: Try Mocking Service

Once the Mocking service is enabled, you can click Try It at the bottom of the RAML viewer to submit an API request and receive the example response you defined. Let’s try to retrieve an Order using the Mock service.

  1. Click Try It at the bottom of the RAML viewer

  1. Enter any value in the order_id text and press send.

  1. Scroll down and you should see a 200 response with the Order record in a JSON format that adheres to the example you set in the API definition.
