Module 5: API Manager

API management is essential to an API-Led architecture as it provides a governance framework to your APIs in all three layers. For API Management to take place, we need to be able to host our APIs, both new and existing, on an API Gateway that will be used for enforcing policies and collecting data for analytics.

MuleSoft can apply governance directly to a MuleSoft implemented API or through a proxy gateway for these and other external existing APIs.

In this lab, we will define an API that will act as a proxy for Mythical Corporation’s Omni Channel API. The proxy will be deployed to Anypoint Platform’s API Gateway, which is powered by the Mule runtime. The Mule run-time can run on-premise, on a private cloud or on the CloudHub service hosted by Anypoint Platform.

For this lab, we are going to configure the API as a proxy to Mythical Corporation’s Omni Channel API Mock Implementation. The API is available as an HTTP Restful API accessible through the base URL http://workshop-omni-channel-mock-service-v40.cloudhub.io/api.

Clients will access the API through the API Gateway which will then forward the requests to the actual Omni Channel API Mock Implementation. Having the proxy deployed on the API gateway allows Anypoint Platform to manage, control access and monitor the usage of the API, which we will look at in the succeeding labs.

API Manager is an API policy management and governance tool that is integrated with the Mule runtime. In this lab, you will utilize the secure connection between the Omni Channel API on runtime manager and API manager to configure policies that will allow us to manage security, quality of service, and compliance policies for your APIs.

In last lab we deployed an Anypoint Gateway to Mythical Corp’s Omni Channel API on CloudHub, now you can:

  • Manage security and enforce policies
  • Manage contracts associated with that API

This makes it easy for the API manager to understand how the API is performing, how it’s being used and by whom, and for the API Administrator to identify potential issues before they arise.

In this lab you will apply policies to your proxy gateway for Mythical Corp’s Omni Channel API. API Manager provides many out-of-the-box (OOTB) policies related to compliance, quality of service and security. In addition, you can add your own policies.

Step 1: Configure your API Proxy

For this lab, we are going to configure the API as a proxy to the API implementation.

  1. Click the API Manager icon to start creating your API
  2. Select SANDBOX as the environment

  1. Now let’s configure an API proxy gateway for this API. Back in the API Administration page, click on Manage API and select Manage API from Exchange.

  1. Configure the API with the following information:
  1. API name: Omni Channel Experience API 
  2. API version: Select the version 1.0.0
  3. Asset version: Select the version 1.0.0
  4. Managing type: Select Endpoint with proxy.
  5. Check the box for managing an API in Mule4 and above
  6. Implementation URI: http://workshop-omni-channel-mock-service-v40.cloudhub.io/api
  7. Proxy deployment target: CloudHub
  8. Path: /

  1. Press Advanced options and configure:
  1. API instance label: <username>-proxy-omni-channel. This Parameter is going to be use to identify the API in the API Manager.
  2. Press Save
  1. After you press Save the deployment configuration section will appear to deploy the proxy.

  1. Configure with the following information:
  1. Runtime version: Select 4.1.4
  2. Proxy application name: <username>-mythical-omni-channel-api-proxy. This property will define the URL and the name of the Mule application
  1. Press Deploy Button.        

  1. Click Close.

  1. Once it is deployed, at the top of the page you will see the API Status. It should be green with a green ball next to it, as shown below. This indicates that your API was successfully deployed and is now being managed.

Step 3: Apply rate limiting policy

To test policy management, you will add a Rate Limiting Policy to the API.

  1. Go to the Omni Channel API Administration page.
  2. Click the Policies tab under the API definitions.

  1. Click the Apply New Policy button.

  1. Click on the Rate Limiting Policy, select the version and Configure Policy.

  1. Enter a maximum number of requests of 3 per 1 Minute as shown and click Apply.
  2. You should see the Rate limiting policy now under Applied Policies


Step 4: Test the API Proxy with policies

Test the API using your browser or Postman.

  1. Access your Proxy URL adding /products/search at the end. For example:

http://<proxy-url>.cloudhub.io/products/search

  1. Test it again 3 times ( reload the browser or push Send button)

  1. On the 3th invocation you will get a message indicating Quota has been exceeded. This demonstrates your Rate Limiting policy has been applied.


Step 5: Remove the "rate limiting" policy

Remove the Rate Limiting Policy after your test by clicking the Remove button.

Step 7 (Optional): Apply rate limiting with SLA

To enforce SLA tiers, you need to apply a rate-limiting or throttling policy that is SLA-based. These policies require all applications that consume your API to register for a specific tier. Their client credentials will be required for each API call so that Anypoint Platform can properly enforce the contracted tier.

Let’s apply a rate-limiting policy to your endpoint.

In order to avoid policy conflict for this lab, ensure there is no policy applied to this API.

  1. Click the Apply New Policy button.

  1. Select the Rate Limiting - SLA Based Policy, and choose the latest version.

  1. Change Client ID Expression value with

 #[attributes.queryParams['client_id']]

  1. Change Client Secret Expression value with

#[attributes.queryParams['client_secret']]

Notice: headers is replaced with queryParams 

In this lab, we will use a different Client ID Expression. This is not required for the policy, but it will make the test easier in the following steps.

  1. Enable Expose Headers
  2. Click Apply to save configuration for the Rate limiting SLA based policy

Step 8: Request access for the API

Remember that the APIs are designed to be discoverable and self served. That’s why we need to go to the Exchange Portal to request access.

  1. Go to the Exchange
  2. Select the Omni Channel Experience API
  3. Press the ellipsis button that is on the right top.

  1. Press Request Access.

  1. In the API Instances, select your API.
  2. In Application, choose Create a new Application
  3. Complete the new application dialog as shown below (you must create a unique application name such as mtm iPhone Application). When finished, click Create
    Since we have tiers associated with our API we also need to select a tier.
  4. Choose the API Instance you deployed
  5. Choose the Trial tier.
  6. Click on the Request API Access button.

  1. By default, all API requests will be approved for the Trial SLA tier. You’ll see your Client ID and Client secret.

You will now test the Omni Channel Experience API. Start using your API:
